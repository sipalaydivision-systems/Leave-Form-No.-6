Figure 1 — Current Flow of Leave Approval (SDO Sipalay City)

Hierarchical flow of leave approval authority and system administration within the Schools Division of Sipalay City.

Leave Approval Chain (Left Branch)

  • SDS — Final approving authority for all leave
  • ASDS — Second-level recommending authority
  • HR Officer — Certifies leave credits and balances
  • AO — First-level approver at school/office level
  • Non-Teaching Personnel — Submit leave applications

IT Administration (Right Branch)

  • IT Administrator — Manages the leave management system
  • Data & Backup Management — Backup, restore, and export data
  • User Account Management — Approve registrations, reset passwords
  • Handles Excel migration, reconciliation, and audit logs

Approval Sequence

  • Employee files leave → AO receives and reviews
  • AO approves → HR certifies leave credits
  • HR certifies → ASDS reviews and recommends
  • ASDS recommends → SDS gives final approval
  • Any level may return application with remarks

Leave Types Supported

  • VL — Vacation Leave (1.25 days/month accrual)
  • SL — Sick Leave (1.25 days/month accrual)
  • FL — Forced/Mandatory Leave (5 days/year)
  • SPL — Special Privilege Leave (3 days/year)
  • WL — Wellness Leave (3 days/year)
  • CTO — Compensatory Time-Off (per Special Order)

User Roles

  • Employee — Files leave, views status and balance
  • AO (Administrative Officer) — 1st-level approver
  • HR Officer — Certifies leave credits
  • ASDS — 2nd-level recommending authority
  • SDS — Final approving authority
  • IT Admin — System administration and maintenance

Technology Stack

  • Backend: Node.js + Express.js
  • Frontend: Vanilla HTML, CSS, JavaScript
  • Storage: JSON flat files (Railway Volume)
  • Email: MailerSend API
  • Authentication: bcrypt + HttpOnly session cookies
  • Deployment: Railway (cloud hosting)

DFD Level 0 — System Overview

Eight major processes and the primary data flows between external entities and the system.

DFD Level 1 — Process 1: Authentication & Registration

Login validation, session management, and new user registration flow.

P1.1 Validate Credentials
Input: Email + password + portal type
Process: bcrypt verify → SHA-256 legacy fallback → rehash if needed
Reads: users.json / *-users.json
Output: Valid/invalid credential result → activity-logs.json
P1.2 Create Session
Input: Valid credentials from P1.1
Process: Generate 96-char hex token, 8-hour expiry
Writes: sessions.json
Output: HttpOnly cookie set on user browser
P1.3 Register New User
Input: Name, email, position, office, portal type
Process: Validate fields → bcrypt hash password
Writes: pending-registrations.json, activity-logs.json
Output: Registration confirmation to user
P1.4 Rehash Password
Input: Legacy SHA-256 hash detected in P1.1
Process: Re-hash with bcrypt (12 rounds)
Writes: Updated hash back to users.json
Output: Account upgraded to bcrypt silently

DFD Level 1 — Process 2 & 3: Leave Submission & Approval Workflow

Complete leave application lifecycle from employee submission through multi-stage approval to final balance deduction.

Process Actor / Input Processing Output / Data Store
P2.1
Validate		 Employee submits: leave type, dates, numDays, signature, SO file Date range check, hours/days calc, duplicate detection Passes to P2.2; error returned to employee on failure
P2.2

P3.1

P3.2

P3.3

DFD Level 1 — Process 4: Monthly Leave Accrual

Automated scheduler that adds 1.25 VL + 1.25 SL every month for non-teaching personnel.

P4.1 Check Accrual Month
Trigger: Server start (daily)
Reads: system-state.json
Output: List of months to process
P4.2 Filter Personnel
Reads: users.json (position field)
Process: Exclude teaching staff
Output: Non-teaching employee list
P4.3 Catch-up Missing Months
Reads: leavecards.json (lastAccrualDate)
Process: Calculate months owed
Output: Cards needing back-credits
P4.4 Add Monthly Credits (+1.25 VL +1.25 SL)
Input: Cards from P4.3
Process: Add 1.25 VL + 1.25 SL per missed month; append transaction record; update lastAccrualDate
Writes: leavecards.json (transactions array), activity-logs.json
P4.5 Update System State
Input: Accrual complete signal
Writes: system-state.json
(lastAccruedMonth, lastAccrualRun)

Entity Summary

Description of each entity, its purpose, and key constraints.

EMPLOYEE_USER

  • DepEd employee accounts
  • PK: id (UUID)
  • UK: email (@deped.gov.ph), employeeNo
  • Password: bcrypt (12 rounds)
  • File: users.json

ADMIN_USER

  • AO / HR / ASDS / SDS / IT accounts
  • PK: id (UUID)
  • UK: email per portal file
  • role: ao | hr | asds | sds | it
  • Files: ao/hr/asds/sds/it-users.json

LEAVE_APPLICATION

  • Core workflow entity
  • PK: id (SDO Sipalay-NN-XXXX)
  • FK: employeeEmail → EMPLOYEE_USER
  • status: pending | approved | rejected | returned
  • File: applications.json

APPROVAL_HISTORY

  • Embedded in LEAVE_APPLICATION
  • One entry per approval stage action
  • Tracks portal, approver, action, timestamp
  • Immutable audit trail

LEAVE_CARD

  • Employee leave balance record
  • PK: employeeId (= email)
  • One card per employee
  • Annual quotas reset yearly (FL/SPL/WL)
  • File: leavecards.json

LEAVE_TRANSACTION

  • Embedded in LEAVE_CARD.transactions[]
  • type: ADD (accrual) | LESS (usage) | LAWOP
  • Running balance per transaction
  • Source tracks whether system or manual

LEAVE_USAGE_HISTORY

  • Embedded in LEAVE_CARD.leaveUsageHistory[]
  • Created on SDS final approval
  • Tracks balance snapshot after each usage
  • Links application to card deduction

CTO_RECORD

  • Compensatory Time-Off grants
  • Multiple records per employee (per Special Order)
  • LIFO deduction order
  • File: cto-records.json

PENDING_REGISTRATION

  • Registration requests awaiting IT approval
  • status: pending | approved | rejected
  • Approved → creates EMPLOYEE_USER or ADMIN_USER
  • File: pending-registrations.json

ACTIVITY_LOG

  • Immutable audit trail of all actions
  • Archived when exceeds 10,000 entries
  • Includes IP address, user agent
  • File: activity-logs.json

SYSTEM_STATE

  • Global singleton for accrual tracking
  • lastAccruedMonth prevents double-accrual
  • Updated after every accrual run
  • File: system-state.json

Relationship Cardinality Summary

All entity relationships with cardinality and business rules.

Cardinality From Entity Symbol To Entity Business Rule
1 : 1 EMPLOYEE_USER ||—|| LEAVE_CARD Each employee has exactly one leave card tracking their balances.
1 : 1 PENDING_REGISTRATION }o—o| USER ACCOUNT An approved registration becomes exactly one employee or admin user.
1 : N EMPLOYEE_USER ||—o{ LEAVE_APPLICATION An employee may submit many leave applications over time.
1 : N EMPLOYEE_USER ||—o{ CTO_RECORD An employee can have multiple CTO grants from different Special Orders.
1 : N LEAVE_APPLICATION ||—o{ APPROVAL_HISTORY Each application accumulates one approval record per workflow stage (up to 5).
1 : N LEAVE_CARD ||—o{ LEAVE_TRANSACTION A leave card accumulates monthly accrual and usage transaction rows.
1 : N LEAVE_CARD ||—o{ LEAVE_USAGE_HISTORY Each SDS-approved leave adds one usage history snapshot to the card.
1 : N EMPLOYEE_USER / ADMIN_USER ||—o{ ACTIVITY_LOG Every user action generates an immutable audit log entry.
M : N CTO_RECORD }o—o{ LEAVE_APPLICATION A CTO record can fund multiple applications; an application can draw from multiple CTO records. Resolved via CTO_APPLICATION_LINK junction.
